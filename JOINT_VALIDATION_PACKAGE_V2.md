# 四组尺寸验证包说明（修正版）

本包用于让用户验证 4 组尺寸，但与上一版不同的是：

- 保留昨天效果较稳定的原联合优化版，作为基线方案
- 保留单目标的 `S 参数最优版` 和 `增益最优版`，用于观察单指标极限
- 移除上一版表现不理想的旧 `联合优化局部版`
- 新增一组基于修正训练流程得到的 `联合优化局部修正版`

这次的重点不是继续冒进，而是围绕昨天已经验证较好的方案，做更稳妥的小步改进。

## 1. 基线联合优化版（建议优先验证）

这组就是用户反馈“昨天效果较好”的基线版本，建议始终保留，作为所有后续改进的对照。

尺寸：

```text
2.57017, 16.8272, 2.97328, 157.277, 77.3126, 1.27523, 90.9747, 77.229
```

预测结果：

- `s11_min_db ≈ -29.1408`
- `gain_max ≈ 1.13158`

对应文件：

- `outputs/new_antenna_model/best_design_conservative.json`
- `outputs/new_antenna_model/best_design_conservative.png`

这组用于观察 `S11` 的最优极限，通常会牺牲较多增益，不建议作为均衡推荐，只建议做参考验证。

对应文件：

- `outputs/new_antenna_model/best_design_s11_only.json`
- `outputs/new_antenna_model/best_design_s11_only.png`

## 3. 增益优先版（保守修正）

这组不再使用旧验证包里过于激进的局部联合方案，而是采用保守搜索后得到的增益优先候选，目标是在不明显恶化 `S11` 的前提下，尝试提升增益。

尺寸：

```text

```

预测结果：

- `s11_min_db ≈ -22.6774`
- `gain_max ≈ 1.19867`

对应文件：

- `outputs/new_antenna_model/best_design_gain_priority_v2.json`
- `outputs/new_antenna_model/best_design_gain_priority_v2.png`

说明：

这组尺寸来自保守型多权重比较结果中的 `gain_priority` 候选，可作为“想抬增益但不想完全放弃 S”的参考方案。

## 4. 联合优化局部修正版

这组是基于修正后的局部训练流程重新生成的方案，不再使用上一版那个 HFSS 实测双降的旧局部结果。

尺寸：

```text
3.775, 15.6312, 1.84656, 153.898, 80.2709, 1.43235, 73.4751, 70.7027
```

预测结果：

- `s11_min_db ≈ -35.0028`
- `gain_max ≈ 1.09919`

对应文件：

- `outputs/new_antenna_model_joint_local_v2/best_design_joint_local_v2.json`
- `outputs/new_antenna_model_joint_local_v2/best_design_joint_local_v2.png`

说明：

这组的意义是验证“修正后的局部模型”是否能在保持较深 `S11` 的同时，避免上一版旧局部方案那种真实仿真双降的问题。

## 建议的验证顺序

建议按下面顺序验证：

1. 先验证第 1 组基线联合优化版，用来确认昨天的较优结果可以稳定复现
2. 再验证第 4 组联合优化局部修正版，判断修正后的局部流程是否真正有效
3. 如果用户更看重增益，再补验证第 3 组增益优先版
4. 第 2 组 `S 参数最优版` 仅用于参考边界，不建议作为主推方案

## 本次修正的原因

上一版验证包中，原第 4 组 `新的联合优化局部版` 在用户 HFSS 验证里出现了：

- `S11` 变浅
- 增益也下降

也就是说，这组方案在真实仿真中没有优于昨天的基线方案，因此不应再作为优先推荐对象。

本次修正版的策略，是回到昨天较好的基线附近做改进，而不是继续依赖已经验证失真的旧局部方案。
